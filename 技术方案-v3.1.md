## 一、项目概述

一个能在实验室/办公室巡逻的AI小车，能听、能看、能感知你的情绪，像一个有情感的AI伙伴。

### 1.1 核心体验

用户视角：**就是跟一个AI正常聊天，这个AI碰巧有身体，能真的动。**

- "你叫什么名字呀" → "我叫小欣呀"

### 2.2 各模块职责

**豆包端到端（语音模块）：**

- 🗣️ 唯一的嘴：所有语音输出都从豆包TTS出
- 👂 唯一的耳朵：所有语音输入都从豆包ASR入
- 💬 闲聊大脑：日常对话豆包自己处理，不经过OpenClaw
- 🏷️ 意图标记：通过system_role让豆包在回复中标记控制意图

**OpenClaw/GLM-4.7（后台大脑）：**

- 🧠 复杂意图判断：视觉查询、状态判断、组合命令
- 🛡️ 安全兜底：速度限制、危险操作拦截
- 📊 结果注入：通过ChatRAGText(502)注入豆包，让豆包自然地说出来
- ⚠️ **不碰喇叭！不触发ChatTTSText（紧急情况除外）**

**Python主程序（协调者）：**

- 🔌 管硬件：串口/摄像头/传感器
- 🔀 意图解析：从豆包回复中提取控制标记
- 🚨 紧急响应：传感器直接触发停车（不等AI）
- 📡 事件注入：传感器/视觉信息通过RAG注入豆包

**STM32（下位机）：**

- 🔄 电机PWM + 编码器PID
- 📏 超声波避障 + IMU姿态
- 🔋 电池电压监测

### 2.3 为什么这样设计？

**为什么豆包当唯一的嘴？**

用户只应该听到一个声音。如果OpenClaw也触发TTS，会出现两个AI抢着说话的割裂感。豆包端到端全双工，自带VAD+打断+自然对话，比我们自己做效果好得多。

**为什么OpenClaw不碰喇叭？**

OpenClaw处理慢（1-3秒），如果等它想完再说话，用户体验很卡。让豆包先回话保持对话流畅，OpenClaw想完结果后通过RAG注入，豆包自然地融入后续对话。

**为什么不用GLM-4-Voice了？**

GLM-4-Voice是半双工，需要自己管VAD/打断/回声消除，复杂且体验差。豆包端到端是全双工，这些都内置了，效果像打电话一样自然。

**为什么还需要OpenClaw？**

豆包不能直接控制硬件（不能调串口、不能读传感器）。OpenClaw提供：

1. 安全约束（速度限制、危险操作拦截）
2. 复杂任务编排（调视觉→分析→决策）
3. 通过skill定义规则，防止模型意外输出

## 三、语音架构（核心，v3.1全新）

### 3.1 豆包端到端语音模块

**连接方式：** WebSocket长连接

**协议格式：** 二进制协议

```
Header: bytes([0x11, 0x14, 0x10, 0x00])  // v1, full-client, json, no-compress
Event ID: 4字节大端整数
Session ID: [4字节长度 + 内容]（可选）
Payload: [4字节长度 + JSON/二进制]
```

**Event ID对照表：**

```
客户端发送:
  1   = StartConnection（建立连接）
  100 = StartSession（创建会话，含system_role）
  101 = FinishSession（结束会话）
  501 = ChatTextQuery（发文字聊天）
  502 = ChatRAGText（中途注入信息）

服务端返回 msg_type:
  9   = 文本（JSON，含content字段）
  11  = 音频（二进制，含OggS标记）
  15  = 错误
```

**实测时序（2026-05-10 Mac mini测试）：**

```
发送"你好呀"后:
  T+0.17s → question_id（豆包收到问题）
  T+0.23s → 第一个结束信号（ASR结束，不是真的结束！）
  T+0.34s → 第一个字"你"
  T+0.38s → "好"
  T+0.42s → "呀"
  T+0.60s → 第一个音频包 ← 从发送到出音频：0.6秒！
  T+0.64s → 第二个音频包（~2KB）
  T+0.72s → 第三个音频包
  ...文本和音频交叉到达...
  T+1.01s → 真正的结束信号

  总计：1.0秒，19个音频包，14.5KB
```

**⚠️ 关键发现：有两个"结束信号"！**

1. 第一个（0.23s）：ASR结束 = "你说完了"，但文本和音频还没来
2. 第二个（1.01s）：Chat结束 = "AI说完了"，这才是真的结束

**音频格式：** OGG Opus，24kHz，单声道

- 每个音频包约2KB（约0.1秒音频）
- 找OggS标记提取

**流式播放方案（边收边播）：**

```
收到第一个音频包(T+0.6s)
  → 立即喂给ffmpeg管道解码
  → ffmpeg输出PCM → PyAudio播放
  → 后续包持续喂，持续播
  → 收到真正的结束信号 → 等播完缓冲 → 停止
```

**延迟：**

- 首字延迟：0.34秒
- 首音频延迟：0.60秒
- 用户说完后0.6秒就能听到AI开口（接近实时对话）

**关键：** Python收到的是豆包的音频"数据"，Python自己决定播不播。Python有完全的播放控制权。

### 3.3 意图解析器

Python端解析豆包回复中的 `[CMD:xxx]` 标记：

```python
import re

CMD_MAP = {
    "move":      ("MOVE", {"speed": 500, "yaw": 0}),
    "stop":      ("STOP", {}),
    "follow":    ("SET_MODE", {"mode": 2}),
    "circle":    ("CIRCLE", {}),
    "speed_up":  ("SPEED_UP", {}),
    "slow_down": ("SLOW_DOWN", {}),
    "patrol":    ("SET_MODE", {"mode": 1}),
}

def parse_intent(doubao_text):
    """从豆包回复中提取命令意图"""
    match = re.search(r'\[CMD:(\w+)\]', doubao_text)
    if match:
        cmd = match.group(1)
        if cmd in CMD_MAP:
            return CMD_MAP[cmd]
    return None  # 无命令，纯聊天
```

**执行流程：**

```
豆包回复："[CMD:move]好嘞马上来~"
  ↓ 正则提取
命令：move → MOVE(speed=500, yaw=0)
  ↓ 去掉标记
显示文本："好嘞马上来~"
  ↓ 同时
执行MOVE指令 → UART → STM32 → 小车开始动
豆包TTS播放"好嘞马上来~" → 扬声器
```

**为什么这样做？**

- ✅ 不需要关键词匹配（豆包理解语义，"靠近点"也能识别为move）
- ✅ 不需要本地模型（豆包已经在想了，零额外成本）
- ✅ 不会漏（"站住""冲""加速"豆包都能理解）
- ✅ 自然（用户感知不到标记的存在）

### 3.4 OpenClaw后台路径

**简单命令（60%+场景）：** 豆包标记 → Python直接执行 → **不经过OpenClaw**

**复杂指令（10%场景）：** 需要OpenClaw判断

```
用户："帮我看看前面有没有人"
  ↓
豆包先回："好的我看看..."（自然过渡，0.5秒）
  ↓
Python检测到这是复杂指令（无CMD标记，但涉及视觉查询）
  ↓
发给OpenClaw → GLM-4.7判断 → 调视觉模块 → 结果："前方2米有1个人"
  ↓
通过ChatRAGText(502)注入豆包："视觉检测结果：前方2米有1个人"
  ↓
豆包自然地说："看到了，前面大概2米的地方有一个人"
```

**闲聊（30%场景）：** 豆包自己回 → OpenClaw完全不管

### 3.5 事件注入（小车主动说话）

Python端的传感器/视觉信息通过豆包API主动注入，让小车"有感知"：

**ChatRAGText(502) — 注入知识（自然，豆包自己组织语言）：**

```python
# 前方有障碍
doubao.send_event(502, {"text": "传感器数据：前方30cm检测到障碍物"})

# 电量低
doubao.send_event(502, {"text": "系统状态：电量剩余15%"})

# 视觉检测到用户情绪
doubao.send_event(502, {"text": "视觉情绪检测：用户表情看起来不太开心"})

# 机器人当前状态
doubao.send_event(502, {"text": "当前状态：跟随模式，目标在前方2米"})
```

豆包收到RAG信息后，会在后续对话中自然地融入，比如主动说"前面有个东西我绕一下"。

**ChatTTSText(500) — 指定文字直接转语音（仅紧急情况用）：**

```python
# 紧急警报（不等豆包组织语言）
doubao.send_event(500, {"text": "小心！前方障碍！"})
```

**UpdateConfig(201) — 运行中更新人设：**

```python
# 根据模式切换性格
doubao.send_event(201, {
    "speaking_style": "你说话简短干练，像保安大叔"  # 巡逻模式
})
```

### 3.6 并发与打断

**问题：ASR文本量大，OpenClaw处理不过来怎么办？**

解决：**简单命令不过OpenClaw**（豆包标记→Python直接执行）。OpenClaw只收到复杂指令，每分钟可能就1-2条，不会被淹没。

**问题：OpenClaw上一次没处理完，新的又来了？**

解决：Python维护 `current_intent` 变量。新意图覆盖旧意图。OpenClaw返回结果时检查是否还是当前意图，不是就丢弃。

**问题：OpenClaw结果返回豆包有延迟？**

解决：

- 简单命令 → 不过OpenClaw，无延迟
- 复杂指令 → 豆包先说"好的我看看"（自然过渡）
- 结果通过RAG注入 → 豆包自然接着说
- 车已经在动了/执行了 → 用户看到效果 → 语音反馈是锦上添花

**问题：豆包正在说话，用户打断了？**

解决：豆包端到端自带 server_vad 模式，自动检测用户说话并打断当前播放。Python端配合：VAD检测到新语音 → 停止当前播放 → 接收新的ASR。

### 3.7 语气/情绪识别

**文字情绪（豆包自带）：**
豆包大模型能从文字推断情绪："你烦不烦啊"→不耐烦，"好开心"→开心

**语气情绪（音调/语速/音量）：**
豆包端到端音频直接进去，理论上能感知语气差异。需要实测验证：

- 用不同语气说同一句话，看豆包回复是否不同
- 如果能感知 → 完美
- 如果不能 → Python本地分析音调/音量 → 通过RAG注入豆包："用户当前情绪：急躁"

**视觉情绪补充：**
摄像头表情识别 → RAG注入豆包 → 豆包综合判断

### 4.4 JY61P IMU

- 型号：维特智能JY61P（基于MPU6050，内置卡尔曼滤波）
- 输出：欧拉角（Roll/Pitch/Yaw），不用四元数
- 万向锁：本项目是地面平台，pitch几乎不到±90°，不受影响
- 通信：软件I2C，PB8(SCL) / PB9(SDA)
- 地址：0x50，读取频率：50Hz
- 作用：偏航角(朝向) + 加速度(碰撞检测) + 航位推算

## 五、底层驱动（STM32端）

### 5.1 麦克拉姆轮逆运动学

STM32收到MOVE指令(speed, yaw)后，分解成4个轮速：

```c
void mecanum_inverse(float vx, float vy, float w,
                     float *v1, float *v2, float *v3, float *v4) {
    float L_plus_W = L + W;  // L=半轴距, W=半纵距
    *v1 = vx - vy - w * L_plus_W;  // 前左
    *v2 = vx + vy + w * L_plus_W;  // 前右
    *v3 = vx + vy - w * L_plus_W;  // 后左
    *v4 = vx - vy + w * L_plus_W;  // 后右
    // 归一化到 -1000~1000
}
```

### 5.2 编码器PID速度闭环

- 3个硬件TIM + 1个软件GPIO读取编码器
- 每个电机独立PID，目标转速由逆运动学给出
- PID输出映射到PWM占空比(0~100%)

### 5.3 超声波避障

- 3个轮流触发（避免串扰），每100ms一轮
- 前方<30cm → 自动减速
- 前方<10cm → 自动停止（即使有前进指令）

### 5.4 IMU数据读取

- 每20ms读取一次（50Hz）
- yaw角：巡逻时航向保持，转弯时精确控制角度
- pitch/roll：检测异常倾斜（翻车保护）

## 六、通信协议

### 6.1 物理层

- USART3：PB10(TX) / PB11(RX)
- 波特率115200，8N1，无流控
- 3.3V电平（树莓派和STM32兼容，无需电平转换）

### 6.2 帧格式

```
┌──────┬──────┬──────┬──────────┬──────┬──────┐
│ 包头  │ 长度  │ 命令  │   数据   │ 校验  │ 包尾  │
│ 0xAA │ 1字节│ 1字节│  N字节   │ 1字节│ 0x55 │
└──────┴──────┴──────┴──────────┴──────┴──────┘

校验和 = 长度 + 命令 + 数据 XOR
```

### 6.3 命令码

**下行（树莓派→STM32）：**

| 命令码  | 名称        | 数据                    | 说明                          |
| ---- | --------- | --------------------- | --------------------------- |
| 0x01 | MOVE      | speed(u16) + yaw(i16) | speed:0~1000, yaw:-180~180° |
| 0x02 | STOP      | 无                     | 紧急停止                        |
| 0x03 | SET_MODE  | mode(u8)              | 0手动/1巡逻/2跟随/3静默             |
| 0x05 | HEARTBEAT | 无                     | 心跳                          |

**上行（STM32→树莓派）：**

| 命令码  | 名称            | 数据                                                | 说明      |
| ---- | ------------- | ------------------------------------------------- | ------- |
| 0x81 | SENSOR_DATA   | dist_L(F)dist_F(F)dist_R(F)battery yaw pitch roll | 传感器10Hz |
| 0x83 | ACK           | cmd(u8)+result(u8)                                | 命令应答    |
| 0xFF | HEARTBEAT_RSP | 无                                                 | 心跳回复    |

### 7.1 视觉方案

### 7.2 视觉进程工作流程

```

    ↓

```

人体太近 → 减速/停止   人体太远 → 加速靠近
人体消失 → 停止等待或切巡逻模式
```

```
视觉情绪 == 语音情绪 → 置信度最高
视觉情绪 != 语音情绪 → 以语音为主
只有视觉 → 用视觉情绪
只有语音 → 用语音情绪
```

### 8.1 进程架构

```
Python主程序（单进程多线程）：
